Spotify Recommender System - Part 4

Creating a Recommender System Using Spotify Data

 Posted on May 7, 2023  |  3 minutes  |  542 words  |  Mujtaba Bader

Project Overview

For this project I’m trying to build a recommender system that will recommend users a selection of artist given certain genres that the user selects. I’d like to make it a web application that users can interact with and not just some code that lies in a github repository. In the previous post I went over preprocessing data and training my model, in this post I’ll discuss the end product, which is a web application.

Web Application

Before diving into creating the web application, I had to decide on what framework I was going to use. I do have some experience with the Flask framework, however I managed to find something that simplified the process alot more, the open-source Python library Streamlit. Streamlit is designed for building data driven web applications, it’s incredibly simple and allows you to develop and deploy your applications in minutes. For my particular project all I needed was a way for users to select genres of interest, select the number of artists they wanted to be recommended, and display the recommended artists. All that was needed was a few lines of code and the application was up and running,

With the web app done all I need to do now is deploy it for users to interact with it. If you’d like to take a look at the code I wrote for this project, you can find it at the github repository.

Project Retrospective

Though I have learned plently from this projecty, overall the end results of the project are probably not too accurate, and I suspect the issues arise as a result of the amount of clusters within our K-Modes model. The data contains over 4000 unique genres and 35000 artists, but we only have 75 clusters in our model. During the training process, we were lead to believe that around 75 clusters would be optimal for our model given the data, and that is in line with how we’d interpret the elbow graph, however if we think beyond these heuristics and just consider exactly what we are trying to do with our data, it does not make sense to bundle this many artists who make distinctly different music into just 75 clusters.

In addition to this, we are not adding new data to our model, so it does not necessairly matter if it fails to generalize to new data, seeing as we only want it to make good recommendations given the inputted genres on our existing data. So one thing I’d change to make this project better is to increase the number of clusters until the recommendations feel they make sense, at the very least no one who likes chicago drill music looking for recommendations should be suggested a k-pop artist.

I’ll be giving an update under this post when I tune the model to give better results, since the elbow method was clearly not the best I’ll opt something like using the silhoutte scores to determine the optimal amount of clusters, aside from that I’ll continue researching other methods and tune it until I feel the recommendations are reasonable.

However, regardless of the accuracy I feel this project has taught me many things and has allowed me to display my technical skills.
